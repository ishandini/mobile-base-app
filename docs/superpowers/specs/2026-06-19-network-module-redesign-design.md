# Network Module Redesign

**Date:** 2026-06-19
**Scope:** `lib/core/network/`, `lib/core/errors/`
**Approach:** Two-phase — Phase 1 correctness, Phase 2 design

---

## Goal

Fix correctness bugs in the networking layer, remove a hidden service-locator
dependency from `APIException`, and improve the `safeApiCall`/`safeRepoCall`
contract so callers have a single consistent Either-based API.

---

## Phase 1 — Correctness

### 1. `APIException` — remove `sl<AppLocalizations>()`

`APIException` becomes a pure data class. The static methods
`_resolveTitle` and `_resolveMessage` (which call `sl<AppLocalizations>()`)
are deleted from this class and their logic moves to `ErrorInterceptor`.

Constructors after change:

- `APIException.initWithResponse(response)` — unchanged; reads `response.message`,
  sets `title = null`. No localization.
- `APIException.initWithException(e)` — simplified; stores `e.toString()` as
  `message`, `title = null`. No localization. Only reached for non-Dio errors
  (generic exceptions) that bypass `ErrorInterceptor`.
- `APIException({required message, required title})` — unchanged.
- `APIException.messageFrom()` static helper — unchanged; no localization.

`_resolveTitle` and `_resolveMessage` static methods are deleted.

### 2. `ErrorInterceptor` — inject `AppLocalizations`, guard `HashValidationException`

Constructor changes to accept `AppLocalizations`:

```dart
class ErrorInterceptor extends Interceptor {
  ErrorInterceptor({required this.localizations});
  final AppLocalizations localizations;
}
```

`onError` logic:

1. If `err.error is HashValidationException` → call `handler.next(err)` and
   return immediately. Preserves the specific exception type; does not wrap it
   into `APIException`.
2. Otherwise → resolve localized title and message from the `DioException`
   (using the moved `_resolveTitle`/`_resolveMessage` logic, now using
   `this.localizations` instead of `sl<>`), create `APIException`, log, and
   re-reject.

`NetworkModule` update: `AppLocalizations` is already a `@lazySingleton`;
inject it into `ErrorInterceptor` via its constructor in the module's
`provideDio()` method (or equivalent factory).

### 3. `SecurityInterceptor` — fix catch block

In `onResponse`, the `catch` block currently rejects with a raw `String`:

```dart
// before
error: 'Hash verification error: $e'

// after
error: HashValidationException('Hash verification error: $e', response: response)
```

This ensures the catch-path produces the same `HashValidationException` type
as the explicit hash-mismatch path, and `ErrorInterceptor`'s guard handles
both correctly.

### 4. Dead branch removal in `APIException._resolveMessage`

Lines 78–81 in `exceptions.dart` contain an unreachable branch:

```dart
} else if (exception.response?.data != null) {
  final baseR = BaseResponse.fromJson(exception.response?.data, (v) {});
  return baseR.message;
}
```

The outer `else if (exception.response != null)` already covers all cases
where `response` is non-null; this inner branch can never execute. It is
deleted as part of the `APIException` simplification above.

## Phase 2 — Design Improvements

### 5. `BaseResponse<T>` → freezed

Replace `@JsonSerializable(genericArgumentFactories: true)` with `@freezed`.

```dart
@freezed
class BaseResponse<T> with _$BaseResponse<T> {
  const BaseResponse._();

  const factory BaseResponse({
    String? status,
    String? message,
    T? data,
  }) = _BaseResponse<T>;

  factory BaseResponse.fromJson(
    Map<String, dynamic> json,
    T Function(Object?) fromJsonT,
  ) => _$BaseResponseFromJson(json, fromJsonT);

  bool get isSuccess => status?.toLowerCase() == 'success';
}
```

Caller impact:
- `response.isSuccess()` → `response.isSuccess` (getter, not method). All
  callers updated.
- Fields become `final`. No callers mutate them; zero functional impact.
- `toJson` is generated by freezed automatically; no manual implementation.
- Run `build_runner` once to regenerate `.freezed.dart` + `.g.dart`.
- Remove `json_serializable` import from `base_response.dart`; freezed drives
  the JSON generation.

### 6. `safeApiCall` — unified Either-based contract

`safeApiCall` changes signature from `Future<T?>` (throw-based) to
`Future<Either<Failure, T>>` (Either-based), matching `safeRepoCall`.

```dart
Future<Either<Failure, T>> safeApiCall<T>(
  Future<BaseResponse<T>> Function() call,
) async {
  try {
    final response = await call();
    if (response.isSuccess) {
      final data = response.data;
      if (data == null) {
        return Left(APIFailure.fromException(
          const APIException(message: 'No data in response', title: null),
        ));
      }
      return Right(data);
    }
    return Left(APIFailure.fromException(APIException.initWithResponse(response)));
  } on DioException catch (e) {
    final ex = e.error is APIException
        ? e.error as APIException
        : APIException.initWithException(e);
    return Left(APIFailure.fromException(ex));
  } on APIException catch (e) {
    return Left(APIFailure.fromException(e));
  } catch (e) {
    return Left(APIFailure.fromException(
      APIException(message: e.toString(), title: null),
    ));
  }
}
```

`safeRepoCall` is unchanged — retained for non-`BaseResponse` datasources
(Firebase Callable Functions, etc.).

Repository call sites collapse from a two-step chain to a single call:

```dart
// before
return safeRepoCall(() async {
  final data = await safeApiCall(() => _apiClient.beneficiaries());
  return data!.toEntity();
});

// after
final result = await safeApiCall(() => _apiClient.beneficiaries());
return result.map((model) => model.toEntity());
```

The `!` bang operator on `safeApiCall` results is eliminated. Null data on a
successful response is now a typed `Left` rather than a silent null.

## What Does Not Change

- `safeRepoCall` — signature and behaviour unchanged
- `APIFailure`, `AppFailure` — unchanged
- `HashValidationException` — unchanged
- `SecurityInterceptor` request hashing — unchanged
- `PgResponse` (Pattern A) — unchanged
- `SapApiClient` — unchanged (no `ErrorInterceptor`, no `safeApiCall`)
- All existing `ApiClient` endpoints — unchanged
- `NetworkModule` Dio configuration — unchanged except `ErrorInterceptor` constructor
